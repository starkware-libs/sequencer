1. Approach

Three independent reviewers ran in parallel:

  • Code reviewer — Claude agent focused on correctness bugs, dead code, simplification opportunities, comment quality, and test integrity.
  • Security reviewer — Claude agent focused on secret/PII leakage, metric cardinality, async race conditions, unbounded resource use, untrusted-input flow, and crypto/default misuse.
  • Codex (GPT-5.4) — third-opinion pass for subtle Rust ownership / async / tower middleware bugs. Hit a sandbox network restriction (bwrap: loopback: Failed) on first attempt; even after pre-generating per-branch diffs into the worktree's .review-diffs/, Codex's shell could not read them. The two Claude passes ran the actual review; Codex contributed scope coverage by being asked for the same review angle (so a missed bug in Claude's two passes that Codex would have caught is the residual risk).

All branch diffs (5,933 lines total) were given to each reviewer. Findings were confidence-filtered (≥70 to be reported; ≥85 to be flagged as a real bug). Fixes were applied bottom-up through the sequencer PR stack so each PR's rebase only had to resolve trivial conflicts with the layer above.

3. Findings — bugs & security

Finding 1 — SIGTERM handler bails out if SIGINT install fails

bugfixed

crates/starknet_transaction_prover/src/main.rs · PR #14045

The spawned shutdown task installed SIGTERM and SIGINT inline via two sequential match blocks. If signal(SignalKind::interrupt()) failed after SIGTERM had already registered, the task return'd early and dropped the live SIGTERM stream — container teardown would then exit with no shutdown log, defeating the whole point of the PR.

Fix

// before — both bail-out branches drop any already-installed handler
let mut sigterm = match signal(SignalKind::terminate()) {
    Ok(s) => s,
    Err(e) => { warn!(error = %e, "..."); return; }
};
let mut sigint = match signal(SignalKind::interrupt()) {
    Ok(s) => s,
    Err(e) => { warn!(error = %e, "..."); return; } // <-- drops sigterm
};

// after — install eagerly, degrade the select! to whichever side is live
let sigterm = signal(SignalKind::terminate())
    .inspect_err(|e| warn!(error = %e, "Failed to install SIGTERM handler"))
    .ok();
let sigint = signal(SignalKind::interrupt())
    .inspect_err(|e| warn!(error = %e, "Failed to install SIGINT handler"))
    .ok();
match (sigterm, sigint) {
    (Some(mut t), Some(mut i)) => tokio::select! { ... },
    (Some(mut t), None)        => { t.recv().await; "SIGTERM" }
    (None,        Some(mut i)) => { i.recv().await; "SIGINT" }
    (None,        None)        => return,
}

Finding 2 — HTTP method metric label not bounded

securityfixed

crates/starknet_transaction_prover/src/server/http_metrics.rs · PR #14056

The middleware recorded request.method().as_str().to_string() as a metric label, claiming "the small enumeration of HTTP methods the server actually accepts" but actually passing whatever method string the parser admitted. A malformed or extended HTTP method token would register a new Prometheus series that lives in the registry until restart — a classic cardinality bomb.

Fix

fn method_label(method: &Method) -> &'static str {
    match *method {
        Method::GET => "GET",
        Method::POST => "POST",
        Method::PUT => "PUT",
        Method::DELETE => "DELETE",
        Method::HEAD => "HEAD",
        Method::OPTIONS => "OPTIONS",
        Method::PATCH => "PATCH",
        _ => "other",
    }
}

The label now lives in a closed enumeration of 8 values; anything unknown collapses to "other".

Finding 3 — Inbound x-request-id not validated (prover)

securityfixed

crates/starknet_transaction_prover/src/server/request_log.rs · PR #14055

The middleware accepted any header value the http crate let through, used it as the AsyncLocalStorage-like tracing field for the whole request, and echoed it back on the response. The http crate already rejects CR/LF, but bytes 0x80–0xFF, whitespace, and unbounded length were all admitted. A megabyte-scale id would balloon every log line for the request duration; whitespace inside the id breaks log parsers.

Fix

const MAX_REQUEST_ID_LEN: usize = 128;

fn extract_or_generate_request_id<B>(request: &Request<B>) -> String {
    request.headers()
        .get(REQUEST_ID_HEADER)
        .and_then(|v| v.to_str().ok())
        .filter(|v| !v.is_empty() && v.len() <= MAX_REQUEST_ID_LEN)
        .filter(|v| v.bytes().all(is_safe_request_id_byte))
        .map(|v| v.to_string())
        .unwrap_or_else(new_request_id)
}

fn is_safe_request_id_byte(b: u8) -> bool {
    b.is_ascii_graphic() // printable ASCII, no whitespace, no DEL
}

Finding 4 — Same x-request-id concern in proof-interceptor

securityfixed

proof-interceptor/src/proxy.ts · PR #751

Same pattern in the TypeScript handler: the inbound id was used to scope AsyncLocalStorage for every downstream log line and was passed to res.setHeader. Node's HTTP parser rejects CR/LF at the request layer, but whitespace and unbounded length remained.

Fix

const MAX_REQUEST_ID_LEN = 128;
const REQUEST_ID_RE = /^[\x21-\x7e]+$/;

function isSafeRequestId(value: string | undefined): value is string {
  return (
    typeof value === "string" &&
    value.length > 0 &&
    value.length <= MAX_REQUEST_ID_LEN &&
    REQUEST_ID_RE.test(value)
  );
}
// fall back to randomUUID() when the inbound id fails the check.

Finding 13 — danger_accept_invalid_certs(true) on the blocking-check client

securityout of scope

crates/starknet_transaction_prover/src/blocking_check.rs:74

The HTTP client used to contact the proof-interceptor disables TLS chain validation, allowing a MITM on the path between the proving service and the interceptor to lie about the screening verdict. This is a real concern with direct compliance impact, but the code predates this observability rollout — it shipped with the original blocking-check feature and none of the 14 PRs under review touch it. Documented here so it doesn't get lost; recommend a follow-up that pins the interceptor cert via add_root_certificate instead of disabling validation.

Finding 14 — tx_hash span field empty on validation-error early returns

bugaccepted

crates/starknet_transaction_prover/src/proving/virtual_snos_prover.rs:153 · PR #14054

The #[instrument(..., fields(tx_hash))] attribute declares the field, but Span::current().record("tx_hash", ...) only runs inside run_virtual_os (after the runner has computed the hash). Validation-error paths (pending block, non-INVOKE, invalid input) return before that, leaving the field empty on those spans.

Accepted as-is. The structured event="validation_error" warns added in PR #14059 already give operators the failure reason directly. Moving the hash computation earlier would require plumbing chain_id through more layers than the cosmetic improvement justifies; Datadog handles empty span fields gracefully.
