# Configuration Labels Hierarchy

The sequencer deployment configurations support a powerful, layered labeling system that allows you to inject Kubernetes labels at various scopes. This system merges labels from the most global definitions down to the most specific object level.

Here are all the possible levels where you can inject labels, ordered from the most global to the most specific:

## 1. Global / Cross-Service Level (`metaLabels`)
Defined in `common.yaml` (or at the root of a service yaml), this applies labels to **every single Kubernetes resource** created for that service (or all services if placed in `common.yaml`).

```yaml
# configs/layouts/hybrid/common.yaml
metaLabels:
  environment: "production"
  app: "sequencer"
```
*Effect:* Appended to Deployments, StatefulSets, Pods, Services, ConfigMaps, Secrets, Ingresses, HPAs, and all other generated resources.

## 2. Pod Level (`podLabels`)
Because `Deployment` and `StatefulSet` controllers generate Pods, you often want labels that apply **only to the Pods** but not to the controller itself (useful for monitoring scrapers, mesh networking, etc.).

You define this at the root of a service yaml (or globally in `common.yaml`):

```yaml
# configs/layouts/hybrid/services/committer.yaml
podLabels:
  scrape-metrics: "true"
  mesh-injected: "true"
```
*Effect:* Merged into the Pod Template Metadata (applied to the actual running pods generated by the controller), but NOT applied to the controller's metadata.

## 3. Construct-Specific Level (`labels` inside a construct block)
Almost every single Kubernetes construct defined in the schema supports its own localized `labels:` dictionary. This will merge the labels **only into that specific Kubernetes object's metadata**.

```yaml
# configs/layouts/hybrid/services/committer.yaml
statefulSet:
  labels:
    component: "database"

service:
  labels:
    routing: "internal"

ingress:
  labels:
    exposure: "public"
```

*Other constructs that support their own isolated `labels:` block:*
- `serviceAccount.labels`
- `rbac.labels`
- `podDisruptionBudget.labels`
- `networkPolicy.labels`
- `priorityClass.labels`
- `gcpPodMonitoring.labels`
- `secret.labels`
- `externalSecret.labels`
- `persistentVolume.labels`

---

## Overlays vs Layouts (The Configuration Dimension)

Because the deployment utilizes a base layout + overlay system, you have an additional dimension of environmental control over where labels are injected:

- **Layout Level:** If you place `metaLabels` in `configs/layouts/hybrid/common.yaml`, it applies to **all deployments** utilizing the hybrid layout base.
- **Overlay Level:** If you place `metaLabels` in `configs/overlays/hybrid/testing/common.yaml`, it will deep-merge with (and override on conflict) the layout labels, applying strictly to the `testing` environment overlay.
- **Service-Overlay Level:** If you place `podLabels` in `configs/overlays/hybrid/testing/services/committer.yaml`, it will apply exclusively to the Pods, strictly for the Committer service, and solely when deploying the testing environment!

## Important Notes & System Behavior
- **Hardcoded Labels Priority:** Construct logic enforces important hardcoded labels (like `service: sequencer-committer` on a service definition). The constructs explicitly merge system-level labels *on top of* user configurations so that fundamental routing relationships cannot accidentally be broken by user label definitions.
- **Root-level schema restriction:** Do not use `labels: {}` at the root of any `.yaml` configuration file. Pydantic validation strictly forbids undefined extra attributes. Always use `metaLabels` for cross-resource labels or `podLabels` for pod-only labels at the root.
