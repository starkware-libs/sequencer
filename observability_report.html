Observability rollout — transaction-prover & proof-interceptor

Date: 2026-05-17  ·  Branches: 14  ·  PRs opened: 14  ·  Tests added: 50+

Source ask: Slack thread — minimum-viable observability for the proof-interceptor and transaction prover that deploy to AWS ECS with Datadog. Discovery-service is owned by someone else.

Hard constraint enforced throughout: never log transaction calldata, signatures, sender addresses, or partner secrets; never log URLs with credentials in userinfo; never expose service state at /health.

1. Approach

Split the wish-list into ten small PRs — five per repo — so each one stays human-reviewable. Started with the highest-value, lowest-risk pieces and ran them in parallel so CI on one could bake while writing the next. Halfway through, ran a confidence-filtered security review across every open branch and caught two critical issues that both got fixed and regression-tested before the report.

Performance overhead

Microbenchmarks live alongside the unit tests and are gated behind --ignored --nocapture (Rust) or RUN_BENCH=1 (TS) so they don't slow normal test runs. Numbers measured on the dev workstation; both services' new code paths sit far below the dominant operations (~10s proving job for the prover, ~50–200ms elliptic-proxy RPC for the interceptor).

PathPer-op costFraction of dominant op
proof-interceptor — logger.info() inside withRequestId~1.1 µs0.005% of one screening RPC
proof-interceptor — empty withRequestId(...) scope (AsyncLocalStorage entry+exit)~1.7 µs0.008%
proof-interceptor — health() probe~0.07 µsfires at most every few seconds; negligible
prover — RequestLogLayer per HTTP request (logging suppressed)see overhead_bench.rs10s+ proving job; sub-µs overhead is invisible
prover — HttpMetricsLayer per HTTP requestsee overhead_bench.rs10s+ proving job; sub-µs overhead is invisible
prover — SaturationMonitor::mark_* (one Mutex<Option<Instant>> lock)~50 nscalled twice per request

All measurements are with the log sink silenced. The actual stdout write cost of an emitted log line dominates the per-call number for any structured log emit — that cost was always paid for the existing per-request log line, so the marginal cost of the new fields (status, request_id) is the JSON-stringify hit on those two extra keys: well under a microsecond.

Concurrency: SaturationMonitor uses a single mutex; with the default max_concurrent_requests = 2 the contention is effectively nil. HttpMetricsLayer's in-flight gauge uses atomics (no lock); metrics-rs counter/histogram updates likewise. Tower layers compose at the type level — no dynamic dispatch in the request path.

export function installProcessCrashHandlers(): void {
  process.on("uncaughtException", (error) => {
    processCrashesTotal.inc({ source: "uncaught_exception" });
    console.error(JSON.stringify({
      event: "uncaught_exception",
      message: error.message,
      stack: error.stack,
    }));
    process.exit(1);
  });
  // ...same shape for unhandledRejection (reason coerced to Error)...
}

PR #751 — structured logger with request-id propagation

What changed

  • New src/logger.ts backed by AsyncLocalStorage. The proxy handler scopes each request with withRequestId(id, ...); every downstream logger.info/warn/error picks up request_id automatically.

proof-interceptor/src/logger.ts 
const context = new AsyncLocalStorage<LogContext>();

function emit(level: LogLevel, fields: Record<string, unknown>): void {
  const requestId = context.getStore()?.requestId;
  const payload: Record<string, unknown> = {
    level,
    ...(requestId !== undefined ? { request_id: requestId } : {}),
    ...fields,
  };
  (level === "error" ? console.error : console.log)(JSON.stringify(payload));
}

proof-interceptor/src/config.ts 
export function redactConfig(config: Config): Record<string, unknown> {
  const { screening, tls, ...rest } = config;
  const redacted: Record<string, unknown> = { ...rest };
  if (screening) {
    const { partnerSecret, ...screeningRest } = screening;
    redacted.screening = {
      ...screeningRest,
      partnerSecret: partnerSecret.length > 0 ? "[REDACTED]" : "[EMPTY]",
    };
  }
  if (tls) redacted.tls = { enabled: true };
  return redacted;
}

[EMPTY] rather than dropping the field — so a missing secret in production is still visible in logs.

PR #754 — /health reflects upstream screening reachability

starknet-privacy CI green

github.com/starkware-libs/starknet-privacy/pull/754 · branch avi/proof-interceptor-health-reachability

What changed

  • New optional health() on TransactionInterceptor. The /health endpoint probes every registered interceptor and returns 503 with the interceptor name + an opaque reason code when any reports unhealthy.
  • ScreeningInterceptor tracks consecutiveFailureStartAt — null while reachable, set on the first failure after a success, cleared on success.
  • When failOpen is on, health() stays green — the operator has accepted shipping transactions through an unreachable upstream.
  • Response body intentionally exposes only the interceptor name and reason code — no timestamps, no error counts, no upstream URL.
proof-interceptor/src/screening-interceptor.ts 
health(): InterceptorHealth {
  if (this.config.failOpen) return { healthy: true };
  if (this.consecutiveFailureStartAt === null) return { healthy: true };
  const unavailableMs = Date.now() - this.consecutiveFailureStartAt;
  if (unavailableMs > this.config.healthMaxUnavailableMs) {
    return { healthy: false, reason: "screening_unreachable" };
  }
  return { healthy: true };
}

Tests: 7 (start state, fail-open stays green, fail-close goes unhealthy after window, recovery on success, /health 503 response, body never leaks internal state).

PR #14045 — global panic hook + graceful SIGTERM shutdown

sequencer CI green

github.com/starkware-libs/sequencer/pull/14045 · branch avi/transaction-prover-panic-shutdown

What changed

  • install_panic_hook registers std::panic::set_hook emitting one structured tracing::error event with location and a forced backtrace.
  • SIGTERM/SIGINT drive a graceful jsonrpsee shutdown. A side task selects on both signals, logs shutdown_started with the signal name, calls ServerHandle::stop, and main logs shutdown_complete once the server drained.
  • Enabled tokio's signal feature in the crate's Cargo.toml.
crates/starknet_transaction_prover/src/main.rs 
let shutdown_handle = server_handle.clone();
tokio::spawn(async move {
    let mut sigterm = signal(SignalKind::terminate())?;
    let mut sigint  = signal(SignalKind::interrupt())?;
    let signal_name = tokio::select! {
        _ = sigterm.recv() => "SIGTERM",
        _ = sigint.recv()  => "SIGINT",
    };
    info!(event = "shutdown_started", signal = signal_name, ...);
    let _ = shutdown_handle.stop();
});
server_handle.stopped().await;
info!(event = "shutdown_complete", ...);

    match self {

        | Self::InvalidTransactionInput(_)
        | Self::ValidationError(_) => outcomes::VALIDATION,

5. Security review pass

After the first four PRs were open, ran a confidence-filtered review focused on secret/PII leakage in logs, metric cardinality, and tower middleware semantics. Two critical findings — both fixed and regression-tested.

Finding 1 — depositor address leaked in JSON-RPC error data. The screening interceptor returned reasons like "address screening: ${address} blocked" and "screening unavailable for ${address}". These flow through proxy.ts into error.data of the JSON-RPC error response, which is sent to the caller. The address is extracted from privacy-pool calldata — so this leaked the depositor to anyone whose transaction was rejected.

// Before:
reason: `address screening: ${address} blocked`,
reason: `screening unavailable for ${address}`,

// After:
if (result === "blocked") return { action: "block", reason: "address_blocked" };
if (result === "unavailable") return { action: "block", reason: "screening_unavailable" };

The e2e test now asserts the opposite of what it asserted before — that the address does not appear in the response payload:

expect(body.error.data).toBe("address_blocked");
expect(body.error.data).not.toContain("0xbad0");

Bundled into PR #753 because it touches the same files.

Finding 2 — credentials leaked in CLI-override log lines. The startup banner in main.rs carefully redacted rpc_node_url via redact_url_host(), but the pre-existing CLI-override info! calls in ServiceConfig::from_args echoed rpc_node_url and blocking_check_url raw. Both can carry credentials in userinfo (https://user:secret@rpc.example.com).

// Before:
info!("CLI override: rpc_node_url: {} -> {}", config.rpc_node_url, rpc_url);

// After:
info!(
    "CLI override: rpc_node_url: {} -> {}",
    redact_url_host(&config.rpc_node_url),
    redact_url_host(&rpc_url),
);

The helper moved out of main.rs into server/log_redact.rs so the startup banner and the CLI-override logs share one implementation. Bundled into PR #14044.
